## 前文回顾

上一篇文章记录了如何用 **Tailscale** 搭建内网穿透，实现对公司 Gitea、禅道和文件服务器的远程访问。

搭建完成后，我顺手试了另一款国内常见的内网穿透工具——**贝锐蒲公英**。本文记录实际使用过程，并对两种方案做系统性对比。

---

## 贝锐蒲公英是什么

蒲公英是贝锐科技（Oray）推出的 SD-WAN 组网产品，和 Tailscale 一样解决「远程访问内网」的问题，但技术路线不同：

| 维度 | Tailscale | 蒲公英 |
|------|-----------|--------|
| 纯软件组网 | ✅ | ✅（也有硬件版本） |
| 核心卖点 | 零配置、WireGuard 加密 | **速度快**、国内服务器、硬件加速 |
| 免费额度 | 3 用户 / 100 设备 | 免费版限制较多，主力是付费版 |
| 加速方式 | 软件打洞 + DERP 中继 | **硬件网关加速**（购买硬件后） |

## 实际使用过程

### 安装与初体验

1. 注册贝锐账号（支持手机号 / 微信登录）
2. 下载蒲公英客户端：https://pgy.oray.com/download/
3. 安装后打开，提示登录

**到这里一切正常。**

### 踩坑：等待 RPC 服务启动失败

登录后客户端弹出提示：

> **「等待 RPC 服务启动失败」**

客户端无法正常进入，反复重启无效。

#### 原因分析

电脑开机后，蒲公英软件自启动速度**比 Windows RPC 服务启动快**，`PgyVPNService` 依赖的 RPC 组件尚未就绪，导致**启动时序冲突**。

#### 解决方法（首选）

1. 按 **`Win + R`**，输入 `services.msc`，回车
2. 找到 **`PgyVPNService`**（蒲公英服务）
3. 右键 → **属性** → **启动类型** 改为 **「自动（延迟启动）」**
4. 点击确定保存
5. **重启电脑** 使设置生效

#### 仍无法解决？再排查以下两项

- **安全软件拦截**：360 / 火绒 / 腾讯管家等可能延迟或拦截蒲公英服务启动。尝试暂时退出安全软件后重试。
- **虚拟网卡冲突**：电脑上若有其他 VPN 软件（TeamViewer、OpenVPN 等）安装的虚拟网卡，可能产生冲突。可到「设备管理器 → 网络适配器」检查多余虚拟网卡，禁用或卸载后重试。

#### 解决结果

按上述方法改为「自动（延迟启动）」并重启后，客户端正常启动，RPC 报错消失。✅

## 蒲公英的内网穿透方式

### 纯软件组网（免费/试用）

和 Tailscale 类似，在每台需要组网的机器上安装客户端，登录同一账号，自动组建虚拟局域网。

```
你的笔记本（外网）
    ↕ 蒲公英隧道（走贝锐国内服务器）
公司装有客户端的机器
    ↕ 访问公司内网服务
```

**优点**：无需购买硬件，免费版可试用，国内服务器延迟低。

**缺点**：免费版限速、限设备数，长期使用需付费。

### 硬件网关加速（付费核心卖点）

这是蒲公英相比 Tailscale **最大的差异化优势**：

在公司网络出口处放置一台**蒲公英路由器（硬件网关）**，它负责：

- NAT 穿透和隧道建立
- 硬件加速加密/解密（减轻客户端 CPU 负担）
- 智能选路（自动选择最优中继节点）

```
你的笔记本（外网）
    ↕ 蒲公英客户端
    ↕ 贝锐国内加速节点（比 Tailscale DERP 快）
    ↕ 蒲公英硬件网关（公司出口）
    ↕ 公司内网服务
```

**实测体验**：在有硬件网关的情况下，速度明显优于 Tailscale 走境外 DERP 中继的场景。

## 蒲公英 vs Tailscale 优劣势对比

| 维度 | 蒲公英 | Tailscale | 胜出方 |
|------|--------|-----------|--------|
| **速度（无硬件）** | 国内服务器，延迟低 | 境外 DERP 中继，延迟高 | 蒲公英 |
| **速度（有硬件）** | 硬件加速，体验最佳 | 仍依赖软件打洞 | 蒲公英 |
| **免费额度** | 免费版限制多（设备数/带宽） | 3 用户/100 设备，完全够个人用 | **Tailscale** |
| **部署成本** | 软件免费，硬件需购买（数百~数千元） | 纯软件，零成本 | **Tailscale** |
| **维护成本** | 低（贝锐托管） | 极低（SaaS 全托管） | 平手 |
| **安全性** | 贝锐自有协议 + AES 加密 | WireGuard（业界标准）+ 自动密钥轮换 | Tailscale（协议更透明） |
| **数据主权** | 控制平面在国内（合规友好） | 控制平面在境外（等保场景需评估） | 视场景而定 |
| **NAT 穿透能力** | 强（国内服务器打洞成功率高） | 极强（多种打洞策略 + DERP 兜底） | Tailscale |
| **跨平台支持** | Windows / macOS / Linux / Android / iOS | 同上 + 路由器 / NAS（更多平台） | Tailscale |
| **适用场景** | 企业需要合规 + 速度，且有预算买硬件 | 个人/小团队，零预算快速组网 | 视需求而定 |

## 选型建议

```
你是……                          推荐方案
─────────────────────────────────────────────────────
个人/小团队，零预算                 → Tailscale（免费版够用）
企业，有预算，对速度要求高           → 蒲公英 + 硬件网关
企业，有合规要求（数据不出境）        → 蒲公英 或 Headscale 自建
出差为主，偶尔访问内网              → Tailscale（5分钟搞定）
固定多地办公室互联                  → 蒲公英硬件方案（稳定性更好）
```

**我的场景**：个人远程访问公司内网，预算为零 → **Tailscale 更合适**。

但如果公司有预算，且对访问速度有较高要求（比如频繁传输大文件），蒲公英硬件方案值得考虑。

## 总结

| 项目 | 结论 |
|------|------|
| 蒲公英客户端体验 | 功能正常，但「RPC 服务启动失败」是个已知坑，需手动改服务启动类型 |
| 速度优势是否属实 | ✅ 属实，国内服务器 + 可选硬件加速，比 Tailscale 走境外 DERP 快 |
| 最大缺陷 | ❌ 免费版限制多，**要获得好体验需要购买硬件设备** |
| 和 Tailscale 的关系 | 不是替代，是**不同预算和场景下的不同选择** |
| 我的选择 | 继续用 Tailscale（零成本），等 UDP 41641 放行后速度会自动提升 |
